## Network Configuration

| VM | OS | Role | IP Address |
|---|---|---|---|
| VM 1 | Ubuntu Server 22.04 | Defender — IDS + Log Forwarder | 192.168.56.10 |
| VM 2 | Kali Linux | Attacker — Attack simulation | 192.168.56.20 |
| Remote | Any (Physical/Cloud) | Splunk Enterprise Server | [Your Splunk IP] |

Both VMs are connected via a **Host-Only Network adapter** in VirtualBox — this isolates attack traffic from your real network while still allowing the Universal Forwarder to reach the remote Splunk server through the host network.

## Data Flow Detail

### Log Files Generated by IDS
```
/var/log/suricata/
    ├── fast.log       → Human-readable one-line alerts
    └── eve.json       → Detailed JSON with full packet metadata (used by Splunk)

/var/log/snort/
    └── alert          → Snort's alert log file
```

### Splunk Universal Forwarder monitors:
```
/var/log/suricata/eve.json     ← Primary source for Splunk
/var/log/suricata/fast.log     ← Secondary human-readable source
```

### Splunk receives, indexes, and makes alerts searchable via SPL queries and dashboard panels.
