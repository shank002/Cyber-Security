# 📊 06 — Splunk Integration (Universal Forwarder + Remote Dashboard)

## Overview

IDS alerts generated by Suricata are exported to a **remote Splunk Enterprise server** using the **Splunk Universal Forwarder (UF)**. The UF monitors the Suricata log files on the Ubuntu VM and continuously ships new log entries to the Splunk indexer over TCP port 9997. Once indexed, alerts are searchable via SPL and visualized on a security monitoring dashboard.

---

## Architecture Recap

```
Ubuntu VM (Defender)                    Remote Splunk Server
────────────────────                    ────────────────────
/var/log/suricata/eve.json
         │
         │  Monitored by
         ▼
Splunk Universal Forwarder   ──────────►  Splunk Enterprise
(inputs.conf + outputs.conf)   TCP 9997   (Indexer + Search Head)
                                          index=ids_alerts
                                          sourcetype=suricata
```

## Part A — Install Splunk Universal Forwarder on Ubuntu VM

### 1. Download the Universal Forwarder

Go to [splunk.com/en_us/download/universal-forwarder.html](https://www.splunk.com/en_us/download/universal-forwarder.html) and download the Linux `.deb` package, or use wget:

```bash
wget -O splunkforwarder.deb "https://download.splunk.com/products/universalforwarder/releases/9.x.x/linux/splunkforwarder-9.x.x-linux-amd64.deb"
```

### 2. Install

```bash
sudo dpkg -i splunkforwarder.deb
```

The forwarder installs to `/opt/splunkforwarder/`.

### 3. Start and Enable on Boot

```bash
sudo /opt/splunkforwarder/bin/splunk start --accept-license
sudo /opt/splunkforwarder/bin/splunk enable boot-start
```

---

## Part B — Configure the Universal Forwarder

### 1. Set the Forwarding Destination (Remote Splunk Server)

```bash
sudo /opt/splunkforwarder/bin/splunk add forward-server <SPLUNK_SERVER_IP>:9997
```

This creates/updates `outputs.conf` at:
```
/opt/splunkforwarder/etc/system/local/outputs.conf
```

Contents of `outputs.conf`:
```ini
[tcpout]
defaultGroup = splunk_indexer

[tcpout:splunk_indexer]
server = <SPLUNK_SERVER_IP>:9997
```

### 2. Configure Log Monitoring (inputs.conf)

Create or edit the inputs configuration:

```bash
sudo nano /opt/splunkforwarder/etc/system/local/inputs.conf
```

```ini
# Monitor Suricata eve.json (primary - rich JSON format for Splunk)
[monitor:///var/log/suricata/eve.json]
disabled = false
index = ids_alerts
sourcetype = suricata
host = ubuntu-ids-lab

# Monitor Suricata fast.log (secondary - human-readable alerts)
[monitor:///var/log/suricata/fast.log]
disabled = false
index = ids_alerts
sourcetype = suricata_fast
host = ubuntu-ids-lab

# Monitor Snort alerts
[monitor:///var/log/snort/alert]
disabled = false
index = ids_alerts
sourcetype = snort
host = ubuntu-ids-lab
```

See [inputs.conf](../configs/inputs.conf) for the full configuration file.

### 3. Restart the Forwarder to Apply Changes

```bash
sudo /opt/splunkforwarder/bin/splunk restart
```

### 4. Verify the Forwarder is Running

```bash
sudo /opt/splunkforwarder/bin/splunk status
```

## Part C — Configure Splunk Enterprise (Remote Server)

### 1. Enable Receiving on the Splunk Server

Log into Splunk Web UI → **Settings → Forwarding and Receiving → Configure Receiving → Add New**

Set the receiving port to `9997`.

Or via CLI on the Splunk server:
```bash
splunk enable listen 9997 -auth admin:password
```

### 2. Create the Index

In Splunk Web UI → **Settings → Indexes → New Index**

```
Index Name: ids_alerts
Max Size:   5 GB (adjust as needed)
```

Or via CLI:
```bash
splunk add index ids_alerts
```

### 3. Verify Data is Arriving

In Splunk Web UI → **Search & Reporting**, run:

```spl
index=ids_alerts | head 10
```

If events appear, the forwarder is working correctly.

## Part D — Configure Sourcetype for eve.json

Suricata's `eve.json` is JSON-formatted. Configure Splunk to parse it properly.

In Splunk Web UI → **Settings → Source Types → New Source Type**:

```
Name:           suricata
Description:    Suricata IDS eve.json alerts
Category:       Network & Security
App Context:    Search & Reporting
Event Breaking: Every line
Timestamp:      Auto
```

Or add to `props.conf` on the Splunk server:
```ini
[suricata]
KV_MODE = json
TIME_PREFIX = "timestamp":"
TIME_FORMAT = %Y-%m-%dT%H:%M:%S.%6N%z
MAX_TIMESTAMP_LOOKAHEAD = 40
SHOULD_LINEMERGE = false
```

## Part E — Splunk Dashboard

The security monitoring dashboard was built in Splunk with the following panels. All SPL queries are documented in detail in:

- [splunk-searches.md](../queries/splunk-searches.md) — Individual SPL queries for investigation
- [splunk-dashboards.md](../queries/splunk-dashboards.md) — Dashboard panel queries

### Dashboard Panels Built

| Panel | Type | Description |
|---|---|---|
| Total Alerts (24h) | Single Value | Count of all alerts in last 24 hours |
| Alerts Over Time | Line Chart | Alert volume trend by 5-minute buckets |
| Alerts by Signature | Bar Chart | Top triggered rule names |
| Top Source IPs | Table | Attacker IPs generating most alerts |
| Alerts by Protocol | Pie Chart | Breakdown of ICMP vs TCP vs UDP alerts |
| Recent Alerts Feed | Table | Live feed of latest 20 alerts |
| SSH Brute Force Events | Table | Filtered view of port 22 attack events |

## Troubleshooting the Forwarder

```bash
# Check forwarder logs for errors
sudo tail -f /opt/splunkforwarder/var/log/splunk/splunkd.log

# Verify monitored files are being read
sudo /opt/splunkforwarder/bin/splunk list monitor

# Test connectivity to Splunk server
telnet <SPLUNK_SERVER_IP> 9997

# Check forwarding status
sudo /opt/splunkforwarder/bin/splunk list forward-server
```
